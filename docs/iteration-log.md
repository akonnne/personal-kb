## 迭代 1：知识图谱跨主题连接（v3 → v4）

### 痛点描述

v3 知识图谱中，不同颜色的主题簇之间完全没有连线。图例显示多个主题，但用户看到的画面是：同色节点内部有连接，但不同颜色之间像被玻璃墙隔开。图看起来像"多个孤岛"，而不是"一张知识图谱"。

**用户反馈原文**："为什么不同颜色之间没有连接了？"

### 量化影响

| 指标 | v3（迭代前） | 期望状态 |
|------|-------------|---------|
| 跨主题边数 | **0 条** | ≥10 条 |
| 跨主题边占比 | **0%** | 约 10–20% |
| 不同主题簇之间的可达性 | **完全不连通** | 至少 1 条路径 |
| 用户满意度 | 用户主动反馈"没有连接" | 无明显抱怨 |

### 根因分析

1. **连接策略只依赖 tag 共现** — 当两张卡片的标签列表完全不同时，无论它们在主题上有多大关联，都不会产生边。
2. **数据本身导致 tag 不跨主题** — 不同策略文档抽取的卡片，标签（如"职业评估"vs"雅思"）天然没有交集。例如：职业评估 card 的 tags = `[职业评估, ACS, EA]`，语言 card 的 tags = `[雅思, PTE, 托福]`，交集为 0。
3. **同主题的 tag 100% 重叠** — 同一主题内的卡片共享完全相同的标签列表，导致组内连接饱和，跨主题连接为 0。

### 实现方案

将连接策略从**纯 tag 共现**改为**tag + topic 混合权重**：

```python
# v3 策略：只基于 tag
weight = len(shared_tags)
# → 跨主题连接：0

# v4 策略：tag * 3 + topic 重叠
weight = len(shared_tags) * 3 + len(shared_topics)
# → 跨主题连接：94 条（占 46%）
```

每个节点保留 **top 3 同主题 + top 2 跨主题**连接，强制建立跨组桥梁。

### 评估效果

| 指标 | v3 | v4 | 变化 |
|------|-----|-----|------|
| 总边数 | 159 | 205 | +29% |
| 跨主题边 | 0 | **94** | **+∞** |
| 跨主题占比 | 0% | **46%** | 显著 |
| 用户后续反馈 | "没有连接" | "布局被拉伸太散"（新问题） |

✅ 跨主题连接问题解决。但引入了新问题：v4 的 94 条跨主题边中，多数是权重很低的弱连接（权重=1），这些弱连接像橡皮筋一样把相距很远的簇硬拉向彼此，导致整个布局被拉伸、分散。这自然引导到迭代 2。

## 迭代 2：布局稳定性（v5 → v6 → v7）

### 痛点描述

v4 虽然解决了跨主题连接，但引入了布局拉伸问题。用户反馈"为什么这么密集"（v3 边太多），然后"布局被拉伸太散"（v4 跨主题边太多）。v5 做了全局稀疏化，但又回到了"不同颜色之间没有连接"的问题。

这是一个**三角不可能问题**：组内连接 + 跨主题桥梁 + 紧凑布局，纯力导向布局无法同时满足三者。

**用户反馈序列**：
- v3: "太密集了，糊成一团" → 稀疏化
- v4: "为什么不同颜色之间没有连接" → 加跨主题边
- v5: "布局被拉伸太散" → 去弱连接
- v6: "怎么还是有问题" → 最终解决方案

### 量化影响

| 版本 | 总边数 | 跨主题边 | 布局紧凑度 | 用户满意度 |
|------|--------|---------|-----------|-----------|
| v3 | 159 | 0 | ✅ 紧凑 | ❌ "簇孤立" |
| v4 | 205 | 94 | ❌ 被拉伸 | ❌ "太散了" |
| v5 | 159 | 0 | ✅ 紧凑 | ❌ "又孤立了" |
| v6 | 162 | 11 | ⚠️ 部分稳定 | ⚠️ "还有问题" |
| **v7** | **165** | **14** | **✅ 紧凑** | **✅ 最终版** |

### 根因分析

1. **纯力导向无法做"聚类+跨接"** — D3 的力导向模拟把所有边同等对待，强连接和弱连接都在拉扯节点，导致布局无法稳定在"簇内紧密、簇间有序"的状态。
2. **弱连接的干扰** — v4 中 94 条跨主题边中有大量权重=1 的弱连接，它们强度低但数量多，整体效果是把不同簇的叶子节点拉向彼此，而非中心。
3. **力导向的冷启动问题** — 节点初始位置随机，前几轮迭代时布局混乱，节点容易漂移到远处后很难回来。

### 实现方案

引入 **聚类引力（Cluster Force）** 替代纯力导向布局：

```python
# v7 布局策略：
# 1. 预定位初始化 —— 每个节点的初始位置围绕主题中心随机分布
#    （半径 10–40px），避免冷启动漂移
# 2. 聚类引力 —— forceX/forceY strength=0.15
#    将节点约束在主题固定区域
# 3. 分级连接 —— 同主题距离 70px/强度 0.12
#                 跨主题距离 200px/强度 0.01（极弱，仅作示意）
# 4. 中心节点桥梁 —— 只连主题中心节点，不连叶子节点
#    避免弱连接把叶子节点拉飞
```

```javascript
// 聚类引力核心
.force("clusterX", d3.forceX(function(d) {
    var c = clusterCenters[d.topic];
    return c ? (width / 2 + c.x * clusterRadius) : width / 2;
}).strength(0.15))
```

7 个主题中心均匀分布在圆上（间隔 51°），节点在各自区域内通过 tag 连接形成子结构，跨主题虚线只连接相邻主题的**中心节点**（共 14 条虚线）。

### 评估效果

| 指标 | v5 | v6 | v7 |
|------|-----|-----|-----|
| 总边数 | 159 | 162 | 165 |
| 跨主题边 | 0 | 11 | **14** |
| 孤立节点 | 0 | 0 | **0** |
| 主题数 | 7 | 7 | **7** |
| 用户反馈 | "没连接" | "还有问题" | **✅ 未再反馈** |

**最终状态**：
- 7 个主题簇均匀分布在圆上，每种颜色独立
- 同主题节点紧密聚集在同一区域
- 跨主题虚线只连接簇中心，不拉飞叶子节点
- 交互式（拖拽、缩放、悬停详情）正常工作

## 迭代方法论总结

### 5 步迭代法的应用

```
痛点发现 → 用户截图/反馈
     ↓
量化影响 → 边数、跨主题数、密集度指标
     ↓
根因分析 → tag 不重叠、力导向局限、弱连接干扰
     ↓
实现方案 → 7 个版本迭代，每次一个假设验证
     ↓
效果评估 → 数据对比 + 用户反馈确认
```

### 关键技术决策

| 决策 | 方案 | 原因 |
|------|------|------|
| DB 引擎 | SQLite+FTS5 | 个人使用，零运维，文件级备份 |
| 抓取策略 | 4 级降级 | 从 markitdown 到基础 HTML，确保不丢内容 |
| 主题分类 | topics.json 语义匹配 | 无需 NLP API，关键词 + AI 理解 |
| 图谱布局 | 聚类引力 | 纯力导向无法兼顾组内紧凑+跨主题连接 |
| 知识蒸馏 | 结构化卡片 | 表格提取 + 要点提取 + 关联计算 |
